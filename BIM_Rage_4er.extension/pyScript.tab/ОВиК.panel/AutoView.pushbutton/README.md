# AutoView - Автоматическое размещение 3D видов на листы (версия 6.0)

## Описание

Скрипт `AutoView_script.py` предназначен для автоматического размещения 3D видов на листы Revit с оптимальным заполнением пространства. Скрипт группирует виды по системам (П-В, ДП-ДВ, А-У) и использует алгоритм упаковки прямоугольников для эффективного размещения на листах. **Ключевое нововведение версии 6.0** — улучшенный трехпроходной алгоритм обнаружения границ с адаптивным порогом, уменьшенное разрешение экспорта для повышения производительности и подробные замеры времени выполнения.

## Основные возможности

- **Автоматическая группировка** 3D видов по префиксам систем (П, В, ДП, ДВ, А, У)
- **Точное измерение размеров** вида с помощью калибровочных квадратов (алгоритм компьютерного зрения)
- **Трехпроходной алгоритм обнаружения границ** с адаптивным порогом на основе медианы яркости (шаг 100px → 25px → 5px)
- **Оптимальная упаковка** видов на листе с использованием алгоритма MaxRects с поддержкой нескольких эвристик (Best Area Fit, Best Short Side Fit, Best Long Side Fit)
- **Гибкие настройки** отступов, зазоров и размещения
- **Интуитивный интерфейс** выбора видов с предварительным просмотром группировки
- **Автоматическое создание листов** с размещенными видами, копированием параметров рамки и типоразмера основной надписи
- **Обработка негабаритных видов** — размещение на отдельных листах
- **Кэширование размеров видов** для повышения производительности
- **Расширенная отладка** с сохранением изображений калибровки и диагностических логов
- **Замеры времени выполнения** ключевых операций (создание листа, экспорт, анализ, удаление)
- **Суммарная статистика** в конце выполнения (время анализа, размещения, общее время)

## Алгоритм калибровки

Для точного определения размеров вида используется следующий процесс:

1. **Создание калибровочного листа** — временный лист с четырьмя эталонными квадратами (5×5 мм) по углам и размещаемым видом в центре.
2. **Экспорт в PNG** с оптимизированным разрешением (1000×1000 пикселей) и настройкой `ZoomFitType.FitToPage`.
3. **Обнаружение квадратов** на изображении с помощью алгоритма поиска границ и морфологических операций (фиксированные координаты).
4. **Вычисление коэффициента калибровки** — фиксированное соотношение 1.5 мм/пиксель (1500 мм / 1000 px).
5. **Трехпроходное обнаружение границ содержимого**:
   - **Проход 1**: шаг 100px по всей области (исключая зоны квадратов)
   - **Проход 2**: шаг 25px в области, расширенной на +100px от результатов прохода 1
   - **Проход 3**: шаг 5px в периметре (±30px от границ прохода 2)
6. **Адаптивный порог** на основе медианы яркости краев изображения:
   - Темный фон (< 100): `is_content = яркость > фон + 20`
   - Светлый фон (≥ 100): `is_content = яркость < фон - 20`
7. **Перевод пиксельных размеров в миллиметры** с добавлением отступов (5 мм).
8. **Проверка качества калибровки** — контрольные пределы для размера квадрата (67–1000 px) и коэффициента мм/пиксель (1.0–2.0).

При неудачной калибровки скрипт сохраняет отладочные изображения и текстовые логи в папке `AutoView_debug` с конкретными рекомендациями по устранению проблем.

## Настройки (константы в коде)

```python
EXPORT_PIXELS = 1000               # Разрешение экспорта (ширина/высота в пикселях)
PADDING_MM = 5                     # Отступ вокруг содержимого вида (мм)
CALIBRATION_SQUARE_MM = 5.0        # Размер эталонного квадрата (мм) - уменьшен с 100 мм
SQUARE_LINE_STEP_MM = 0.5          # Шаг линий вложенных квадратов (мм)
SQUARE_LINE_COUNT = 5              # Количество вложенных квадратов
MM_PER_PX = 1.5                    # Фиксированный коэффициент мм/пиксель (1500 мм / 1000 px)
MIN_SQUARE_SIZE_PX = 67            # Минимальный ожидаемый размер квадрата (px) - 100 мм × 0.67 px/мм
MAX_SQUARE_SIZE_PX = 1000          # Максимальный ожидаемый размер квадрата (px)
MIN_MM_PER_PX = 1.0                # Минимальный коэффициент мм/пиксель
MAX_MM_PER_PX = 2.0                # Максимальный коэффициент мм/пиксель
VIEW_SIZE_CACHE = {}               # Кэш размеров видов (повышение производительности)
```

## Инструкция по использованию

1. **Подготовка**  
   - Убедитесь, что в проекте есть 3D виды с корректными именами (префиксы систем: П, В, ДП, ДВ, А, У).
   - Создайте лист-шаблон с рамкой (титульным блоком), который будет использоваться как образец.

2. **Запуск**  
   - В pyRevit нажмите кнопку «AutoView» на панели «TEST».
   - Выберите лист-шаблон в диалоговом окне.

3. **Выбор видов**  
   - В открывшемся интерфейсе отметьте виды, которые нужно разместить.
   - Используйте кнопки «Выбрать все» / «Снять выделение» для удобства.
   - Просмотрите предварительную группировку видов по системам.

4. **Выполнение**  
   - Нажмите «Разместить виды».
   - Скрипт выполнит анализ размеров каждого вида (с калибровкой), упаковку на листы и создание новых листов с размещенными видовыми экранами.
   - В консоли будет выведен подробный лог процесса с замерами времени выполнения каждой операции.

5. **Результат**  
   - Новые листы будут созданы с номерами, производными от номера шаблона (например, «101_П-В», «102_ДП-ДВ»).
   - Виды будут размещены внутри рамки с учётом отступов и зазоров.
   - Папка `AutoView_debug` (рядом со скриптом) будет содержать отладочные изображения и логи при необходимости.
   - В конце выполнения будет показана статистика времени: анализ, размещение и общее время.

## Поддерживаемые префиксы систем

| Группа | Префиксы | Описание |
|--------|----------|----------|
| П-В | П, ПЕ, В, ВЕ | Приточные и вытяжные системы |
| ДП-ДВ | ДП, ДПЕ, ДВ, ДВЕ | Дополнительные приточные и вытяжные системы |
| А-У | А, У | Аварийные и удаляемые системы |

## Параметры рамки листа

Скрипт автоматически определяет размеры рамки листа (титульного блока). Приоритет использования параметров:
1. **Высота_Реальная** и **Ширина_Реальная** (пользовательские параметры семейства рамки)
2. **Высота** и **Ширина** (альтернативные имена параметров)
3. **Bounding Box** титульного блока (геометрические габариты)

Если в вашем проекте используются другие имена параметров, отредактируйте функцию `get_sheet_frame_mm` в коде.

## Отладка и диагностика

При возникновении ошибок калибровки или размещения скрипт сохраняет подробную информацию в папке `AutoView_debug`:
- **Изображения калибровки** с отмеченными квадратами и границами содержимого (префиксы `cal_`, `OK_`, `FAIL_`)
- **Текстовые логи ошибок** с описанием проблемы и рекомендациями (префикс `ERROR_`)
- **Визуальная отладка** алгоритма упаковки (при необходимости)
- **Улучшенные изображения** после обработки (префикс `enhanced_`)

Рекомендуется периодически очищать эту папку, чтобы избежать накопления файлов.

## Производительность и оптимизация

Версия 6.0 включает значительные улучшения производительности:

1. **Уменьшение разрешения экспорта** с 2000 до 1000 пикселей (в 2 раза меньше данных)
2. **Трехпроходной алгоритм** сканирует ~350 пикселей вместо 6400 (в 18 раз меньше операций)
3. **Замеры времени** для профилирования ключевых операций:
   - Создание калибровочного листа
   - Экспорт изображения
   - Анализ изображения (enhance + трехпроходной поиск)
   - Удаление временных объектов
4. **Суммарная статистика** в конце выполнения:
   - Итоговое время анализа всех видов
   - Итоговое время размещения на листах
   - Общее время выполнения скрипта

Ожидаемая точность определения границ: ±4 мм (5 px × 1.5 мм/px + отступы).

## Известные ограничения

- Скрипт работает только с **3D видами**.
- Калибровочные квадраты должны быть хорошо видны на экспортированном изображении (достаточный контраст, отсутствие перекрытий).
- При очень больших количествах видов (десятки) время выполнения может увеличиться из-за экспорта изображений (используется кэширование для ускорения повторных запусков).
- Алгоритм обнаружения границ оптимизирован для разрешения 1000×1000 пикселей; при изменении разрешения требуется корректировка констант.

## Планы на будущее: оптимизация для Г-образных схем

Для Г-образных схем можно сэкономить **20-40% площади** на листе, размещая маленькие виды в пустотах!

### Что нужно для реализации (теория)

#### 1. Разбиение bounding box на части в `find_content_bounds_3pass`

После нахождения общего bounding box, делается **4-й проход** — поиск пустот внутри:

```python
# После прохода 3 (уточнение границ):
# Проход 4: поиск пустот внутри bounding box
# Сканируем внутренность найденного прямоугольника
# Ищем строки/столбцы БЕЗ содержимого
# Если находим сплошную пустую полосу — делим прямоугольник на части
```

#### 2. Структура данных для составного вида

```python
# Вместо одного размера:
view_sizes[view.Id] = (width_mm, height_mm)

# Возвращаем список частей:
view_sizes[view.Id] = {
    'parts': [
        {'w': 100, 'h': 300, 'rel_x': 0, 'rel_y': 0},    # левая часть
        {'w': 200, 'h': 100, 'rel_x': 0, 'rel_y': -100},  # нижняя часть
    ],
    'bbox': (200, 300),  # общий bounding box
}
```

#### 3. Модификация упаковщика

При размещении составного вида:
- Первая часть размещается как обычно
- Остальные части получают координаты = `(x + rel_x, y + rel_y)`
- Все части помечаются как занятые (для других видов)
- Другие виды могут занимать пустоты **внутри** составного bounding box

#### 4. Viewport.Create

Всё равно создаётся **один** видовой экран по **общему bounding box**. Но для упаковщика доступная площадь увеличивается за счёт пустот.

### Что это даст

| Сценарий | Сейчас | С разбивкой |
|----------|--------|-------------|
| Г-образная схема 200×300 | Занимает 200×300 = 60 000 мм² | R1: 100×300 + R2: 200×100, пустота 100×200 доступна |
| Маленький вид 80×120 | Не влезает | Влезает в пустоту 100×200 ✅ |
| Экономия площади | 0% | **30-40%** для Г-образных схем |

### План реализации

1. **Проход 4** в `find_content_bounds_3pass` — поиск пустот
2. **Разбиение** bounding box на прямоугольники
3. **Модификация упаковщика** — поддержка составных объектов
4. **Тестирование** на реальных схемах

Готово для реализации когда понадобится! Сейчас скрипт работает с обычными прямоугольниками, и этого достаточно для большинства схем.
